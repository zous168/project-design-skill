# CRM 工程 / 目录结构设计

> 对应 skill 文档 **10 — Project Structure & Directory Design**。
> 目录结构由**技术选型 + 架构形态**决定，故先锁定技术方向。

## 技术方向（Phase 0 决策）

| 维度 | 选择 | 影响 |
|------|------|------|
| 前端 | **Vue 3 + Element Plus + Vite + Pinia + Vue Router** | 决定前端目录与组件风格（原型已采用 EP 视觉语言） |
| 后端 | **Node.js + NestJS** | 模块化、装饰器、DI；目录天然按模块组织 |
| 架构形态 | **模块化单体** | 单一可部署单元，内部按销售域模块清晰隔离，契合“本地私有化 / 服务器集中部署” |
| 仓库 | **pnpm monorepo** | 前后端 + 共享契约同仓，类型/DTO 复用 |

---

## 仓库总览（monorepo）

```
crm/
├── pnpm-workspace.yaml
├── packages/
│   ├── web/            # 前端 Vue 3 + Element Plus
│   ├── server/         # 后端 NestJS（模块化单体）
│   └── contracts/      # 前后端共享：DTO / 枚举 / OpenAPI 类型
└── docs/               # 本设计文档集
```

> 共享 `contracts/` 让线索、商机、跟进等 DTO 与枚举（如商机阶段、线索状态）**前后端单一定义**，避免字段漂移——直接呼应一致性审核中“数据/接口对齐”。

## 模块边界与依赖规则（模块化单体的关键）

1. **域模块之间不直接 import 对方内部**——经各自 `*.service` 暴露的接口或领域事件协作（如赢单 → 触发事件 → integration 适配器通知 CSM/Hermes）。
2. **外部系统只经 `integration/` 适配器**访问，域模块不直连 Hermes/客户全景/CSM，便于私有化部署时替换/降级。
3. **`contracts/` 为唯一契约源**，前后端共享，禁止各自重复定义 DTO/枚举。
4. **演进路径**：模块边界即未来微服务的切分线——若某模块（如话术 RAG）需独立扩展，可低成本拆出，无需重构调用方。

## 命名约定

- 文件：`kebab-case`；类：`PascalCase`；NestJS `*.controller/service/module/dto/entity`。
- 前端组件 `PascalCase.vue`；composable `useXxx`；Pinia store 按域单数命名。
- 接口路径：`/api/v1/<module>`；写操作带 `requestId` 幂等（呼应需求 §5.1）。

# CRM 技术选型与评估

> 全闭环产出 · 对应 skill 文档 **09**。headline stack 已在 Phase 0 选定(本文**正式论证**并定细化项)。上游:[03 架构](03-系统架构.md)、[07 异步](07-异步处理.md)、[08 NFR](08-非功能性需求.md)。

## 1. 已定 headline stack(Phase 0)
| 领域 | 选择 |
|------|------|
| 前端 | Vue 3 + Element Plus + Vite + Pinia + vue-i18n |
| 后端 | Node.js + NestJS |
| 数据库 | **PostgreSQL**(cfg/data/logs/stats 同库分 schema)|
| 缓存/队列 | **Redis** + BullMQ |
| 架构形态 | 模块化单体 |

## 2. 决策评估(每项含备选 + 理由 + 取舍)

### 2.1 主数据库 — PostgreSQL
| 选项 | 拟合 | 事务/关联 | 灵活性 | 运维/私有化 | 结论 |
|------|------|-----------|--------|-------------|------|
| **PostgreSQL** | 高 | 强(ACID/外键) | JSONB | 成熟、易私有化 | ✅ 采用 |
| MySQL | 高 | 强 | JSON 弱 | 普及 | 备选 |
| MongoDB | 中 | 弱(跨文档事务受限) | 高 | — | ✗(销售域强关联+折扣/赢单事务) |
- **理由:** 销售域强关联 + 折扣/赢单事务需 ACID;cfg/data/logs/stats 以 **schema** 隔离于单实例,部署最简、后续可平滑拆库。**取舍:** 暂不引入独立分析库,换取私有化部署简单;大数据量时 stats 可迁 ClickHouse(已在 [05](05-数据设计.md) 留演进位)。

### 2.2 缓存 / 队列 — Redis + BullMQ
- **理由:** 单依赖同时支撑 BullMQ 异步/调度队列(见 [07](07-异步处理.md))+ 查重/话术热点缓存(命中 [08 NFR](08-非功能性需求.md) P95≤200ms)。**取舍:** 未上 Kafka/RabbitMQ,换取轻量与私有化内置;超大吞吐时队列可替换为 RabbitMQ(适配器隔离)。

### 2.3 后端 — NestJS / 前端 — Vue 3 + Element Plus
- 后端 NestJS:模块化/DI 天然契合模块化单体与域模块边界;`@nestjs/swagger` 直出 OpenAPI(见 [06](06-接口设计.md))。
- 前端 Vue 3 + Element Plus:中后台表格/表单/弹窗组件齐全;EP 暗色 + vue-i18n 支撑主题/双语默认([08 NFR](08-非功能性需求.md))。

## 3. 细化项(本文新定)
| 项 | 选择 | 理由 |
|----|------|------|
| ORM | Prisma / TypeORM(NestJS 生态) | 迁移 + 类型安全 |
| OpenAPI | `@nestjs/swagger` → `/api/docs` | 契约源(见 06) |
| 鉴权 | JWT(工作台)/ API Key(开放 API) | 见 03 横切 |
| 调度 | BullMQ repeatable / k8s CronJob | 见 07 |
| 部署 | 容器化,私有化(docker-compose)/ 集中(k8s) | 见 03 部署 |

## 4. 整体栈与接受的权衡
单 PG + Redis 起步、模块化单体——**优先简单可私有化**,以"单库瓶颈/单体扩展性"为已知风险(见 [11 风险评估](11-风险评估.md)),并预留拆库/拆服务的演进路径(模块边界 = 切分线)。

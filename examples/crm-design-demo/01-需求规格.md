# CRM 需求规格说明书（PRD）

> 全闭环产出 · 对应 skill 文档 **01**。输入基线:用户《CRM 需求规格说明书 v3》。
> 配套:[02 领域分析](02-领域分析.md) · [03 系统架构](03-系统架构.md) · [04 功能树](04-功能树.md) · [06 接口设计](06-接口设计.md)

## 1. 背景与问题
全渠道销售体系下,留资分散在官网/微信/企微/抖音/广告,人工承接导致**重复跟进、撞单、转化慢、话术不统一**。现状:线索靠人工分配、跟进无标准 SOP、销售话术散落个人。CRM 作为**销售域业务数据底座与规则管控中心**,统一线索入口、标准化商机与跟进、沉淀话术与报价规则,并为 Hermes(生成式 AI 出口)提供结构化数据与规则边界。

## 2. 目标与成功指标(可度量)
| 目标 | 指标 | 现状→目标 |
|------|------|-----------|
| 统一线索入口、消除撞单 | 撞单率 | 8% → **<1%** |
| 标准化培育、提升转化 | 线索→商机转化率 | 42.8% → **≥60%** |
| 加速成交 | 平均跟进周期 | 5.8 天 → **≤4.5 天** |
| 话术赋能 | RAG 话术命中率 | 64% → **≥85%** |
| 售前-售后闭环 | 赢单→CSM 开通时延 | 人工 → **≤1 分钟(自动)** |

## 3. 目标用户 / 角色
| 角色 | 代号 | 画像与核心诉求 |
|------|------|----------------|
| 销售坐席 | R1 | 高效跟进本人线索/商机、登记跟进、申请特价 |
| 销售主管 | R2 | 团队看板、审批特价、改派、关闭;含 R1(团队范围) |
| 运营管理员 | R3 | 配置 SOP/话术/角色/渠道分配规则、看报表;不直接销售 |
| 集成账号(Hermes) | R4 | 机器账号:建线索、回写跟进、查 SOP、检索话术、赢单回调 |

## 4. 范围
- **In:** 留资承接、线索分配、商机阶段跟进、报价/折扣管控、话术与 SOP 沉淀、销售人工兜底、运营统计、对 Hermes/客户全景/CSM 的接口。
- **Out:** 售后工单/投诉(→ CSM)、订单明细/物流(→ 订单域)、客户主数据(→ 客户全景域,本系统仅持 `customerId`)、生成式应答(→ Hermes)。

## 6. 关键用户故事(验收)
- **US-1(R4 Hermes 建线索):** 抖音留资进线 → 调 `POST /api/v1/leads`(幂等)。**验收:** 同号 5 分钟内重复进线返回同一 `leadId` 且 `deduped=true`,不新建。
- **US-2(R1 推进赢单):** 谈判通过 → 推进至赢单。**验收:** 触发客户全景回写、CSM 通知、Hermes 回调三者均产生记录(可在 logs 查 traceId)。
- **US-3(R2 审批特价):** 坐席申请 8.5 折(<9 折)。**验收:** 系统拦截并生成审批单,主管通过后方可报价;全程写 op_log。

## 8. 假设 / 依赖 / 约束
- 假设:客户主数据由客户全景域提供 `customerId`;Hermes 为唯一生成式出口。
- 依赖:客户全景 `identify`/`behavior.sync`、CSM 售后事件、Hermes 工具插件。
- 约束:本地私有化 / 服务器集中部署;敏感联系方式脱敏 + 操作审计。
